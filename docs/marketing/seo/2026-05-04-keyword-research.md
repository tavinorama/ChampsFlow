## 4. Pre-Launch Technical SEO Checklist

Complete before any page goes live. Mark each item DONE in a separate tracking sheet.

**Title Tags**
- [ ] Every page has a unique title tag, 50ŌĆō60 characters, keyword-first where natural
- [ ] Homepage title pattern: `Organic Posts ŌĆö AI Social Media Scheduler for Small Business`
- [ ] Feature pages: `[Feature Name] | Organic Posts`
- [ ] Blog posts: `[Post Title] | Organic Posts Blog`
- [ ] No title tag duplication across pages

**Meta Descriptions**
- [ ] Every page has a unique meta description, 140ŌĆō155 characters
- [ ] Descriptions include a call to action and primary keyword naturally
- [ ] No auto-generated or blank descriptions

**Heading Hierarchy**
- [ ] One H1 per page; H1 matches or closely mirrors the title tag keyword intent
- [ ] H2s structure the main page sections; no skipped heading levels (H1 ŌåÆ H3 without H2)
- [ ] Blog posts: H2/H3 match FAQ or long-tail question variants for featured snippet eligibility

**Canonical Tags**
- [ ] `<link rel="canonical" href="[absolute URL]" />` on every page
- [ ] www vs. non-www consistent; canonical points to preferred version
- [ ] No self-referencing canonical errors on paginated content

**Sitemap**
- [ ] `sitemap.xml` generated and submitted to Google Search Console before launch
- [ ] Sitemap includes all indexable pages; excludes `/auth`, `/account`, `/billing`, `/onboarding` (already behind auth)
- [ ] Sitemap referenced in `robots.txt`

**robots.txt**
- [ ] `robots.txt` live at root domain
- [ ] Disallow: `/auth/`, `/account/`, `/onboarding/`, `/billing/`, any staging subpaths
- [ ] Sitemap URL declared: `Sitemap: https://[domain]/sitemap.xml`

**JSON-LD Structured Data (schema.org)**
- [ ] `SoftwareApplication` schema on homepage and feature pages:
  ```json
  {
    "@context": "https://schema.org",
    "@type": "SoftwareApplication",
    "name": "Organic Posts",
    "applicationCategory": "BusinessApplication",
    "operatingSystem": "Web",
    "offers": {
      "@type": "Offer",
      "priceCurrency": "USD",
      "price": "0",
      "description": "Free trial available"
    }
  }
  ```
- [ ] `FAQPage` schema on blog posts with FAQ sections (boosts featured snippet eligibility)
- [ ] `Article` schema on all blog posts (datePublished, dateModified, author)
- [ ] Validate all schema via Google Rich Results Test before launch

**Open Graph Tags**
- [ ] `og:title`, `og:description`, `og:image` (1200x630px), `og:url`, `og:type` on every page
- [ ] `twitter:card`, `twitter:title`, `twitter:description`, `twitter:image` for X sharing
- [ ] OG image is brand-consistent and loads without redirect

**hreflang**
- [ ] `hreflang="en-US"` and `hreflang="en-GB"` tags on all pages (EU English market + US market)
- [ ] `hreflang="x-default"` pointing to en-US variant as fallback
- [ ] Reciprocal hreflang tags confirmed (each locale page references all others)
- [ ] Implement via `<link>` in `<head>` (not sitemap, to avoid sync lag)

**Core Web Vitals**
- [ ] LCP target: under 2.5s ŌĆö audit hero image and largest above-fold element; use next-gen formats (WebP/AVIF)
- [ ] CLS target: under 0.1 ŌĆö reserve space for images/embeds; no layout shift on cookie banner injection
- [ ] INP target: under 200ms ŌĆö audit any heavy JS bundles on marketing pages; defer non-critical scripts
- [ ] Run PageSpeed Insights on homepage, `/pricing`, and one blog post before launch

## 6. Analytics + Cookie Consent Note

If GA4 is implemented for pre-launch traffic tracking:

**GDPR requirement (mandatory)**: The cookie consent banner documented in `docs/04-ux.md` ┬¦6 must fire before any GA4 tracking script initialises. GA4 must be loaded conditionally ŌĆö only after explicit user consent is recorded. Do not use `gtag.js` in a way that sends events or sets cookies on page load before the consent signal is captured.

Implementation pattern: use GA4's Consent Mode v2 with `analytics_storage: 'denied'` as the default state. Only update to `'granted'` after the user accepts analytics cookies in the consent banner. This applies to both EU and US visitors (California CCPA requires opt-out parity; defaulting to denied simplifies compliance across jurisdictions).

Privacy-preserving alternatives that do not require cookie consent banners under GDPR (because they do not use cookies or process personal data): Plausible Analytics, Fathom Analytics, Umami (self-hosted). These are GDPR-compliant by design and may be preferable for the EU-first segment, particularly during the pre-launch phase where consent infrastructure may not be fully tested.

Any recommendation to implement GA4 must be reviewed by `legal-privacy-officer` before deployment to confirm DPA and data transfer mechanisms (EU-US Standard Contractual Clauses) are in place.
