## ✅ HIGH — Runtime tenant isolation — **REMEDIATED 2026-06-18**
**Original finding:** RLS and append-only were correctly DEFINED in migrations
(FORCE ROW LEVEL SECURITY; REVOKE UPDATE/DELETE on `app_user`), but the app
connected to Postgres as the **superuser** and never `SET ROLE app_user`.
Superusers bypass RLS/REVOKE, so tenant isolation was inert at runtime; tenant
context also used `set_config(local)` outside a transaction, so it reverted
before the next query.

**Fix shipped (this surface = the API request path):**
1. `apps/api/src/db/tenant-context.ts` — an AsyncLocalStorage tenant scope.
   `requireAuth` runs every authenticated, non-super-admin request inside it
   (super-admins stay unscoped for cross-tenant ops by design).
2. `apps/api/src/db/client.ts` — `query()` now wraps each tenant-scoped query in
   a transaction that `set_config('app.current_tenant_id', …)` **and**
   `set_config('role', 'app_user', …)`. Dropping into the non-superuser role is
   what *activates* FORCE RLS; the transaction makes both LOCAL settings hold and
   reset cleanly on the pooled connection.
3. Migration `20260618000001` — adds the missing `app_user` GRANT on
   `ccpa_requests` and grants `app_user` membership to the login role so
   `SET ROLE` works on managed/Supabase deploys.
4. **Boot guard** (`assertAppDbRoleSafe`) — the API refuses to start if
   `APP_DB_ROLE` is missing, is a superuser/BYPASSRLS role, or can't be assumed
   (verified live: `APP_DB_ROLE=postgres` → `rls_role_assertion_failed` →
   shutdown). RLS can no longer be silently disabled.
5. `check-rls.sql` already covers all 23 tables (earlier fix).
6. `tests/integration/db/rls.test.ts` — **rewritten** as a real two-tenant proof
   (insert rows for A and B; as scoped `app_user`, cross-tenant SELECT/UPDATE/
   DELETE all return 0 rows; same-tenant works; superuser CONTROL sees both;
   append-only REVOKE survives the role drop). **21 tests pass** against local pg.
7. Migration `20260618000002` — tightened `dsr_requests`/`ccpa_requests` RLS so
   NULL-tenant (intake) rows are no longer visible to a scoped tenant
   (adversarial-review finding; regression-tested).

Defense-in-depth `WHERE tenant_id` filters were also added to the unqualified
`provider_keys` SELECT/DELETE. JWT `tenant_id` is now UUID-validated in
`requireAuth`.

**Residual (tracked, out of scope for this fix):**
- The BullMQ **worker** still queries via raw postgres-js as the login role
  (relies on explicit `WHERE tenant_id`). Apply the same role-drop pattern there.
- The **DSR erasure cascade** (`dsr.ts`, super-admin/unscoped) is a sequence of
  autocommitted statements — not atomic. Pre-existing; not caused by this change;
  should be wrapped in one transaction for GDPR Art. 17 integrity.

## 🟠 MEDIUM (code)
- **SERP/AIO probe is a stub in live mode** — `providers/serp.ts` throws when
  `SERP_API_KEY` is set, so Google AI Overview presence (`aioPresence`) silently
  stays false in production (off-site/Reddit have their own DataForSEO calls and
  do work). Fix: implement the live DataForSEO SERP call, or drop SERP from
  REQUESTED_PROVIDERS so it doesn't hit failedProviders on every keyed audit.
- **Coverage gaps on live pipeline modules** — site-crawl.ts, offsite-signal.ts,
  content-geo.ts (all run on every audit) have **zero unit tests**; the
  ssrf-guard redirect/DNS runtime path is ~62–69% covered and not individually
  gated. Add unit tests (esp. SSRF redirect-to-internal/DNS-rebinding cases).
- **Dead code:** `scoring.ts buildStrategyPromptInput` (the GEO-A2 anonymizer) has
  zero production call sites. GEO-A2 still holds (ProbeQuery only carries the
  client's own brand; competitor names never sent), but this function is not the
  enforcer it appears to be — wire it in or remove it.
