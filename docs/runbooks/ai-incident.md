# AI Incident Response Runbook (A4 + L-UX-2)

> Owner: devops-engineer + product-manager · Created: 2026-05-11

## TL;DR

This runbook activates when the `draft_reported` Grafana alert fires (>5 reports per `generation_id` or >20 reports per day). It covers the investigation, mitigation, customer communication, and escalation path.

## Alert Thresholds

| Metric | Alert threshold | Grafana alert name |
|---|---|---|
| `draft_reported` per `generation_id` | > 5 reports | `ai-incident-generation-threshold` |
| `draft_reported` total per day | > 20 reports | `ai-incident-daily-threshold` |
| `regen_to_approval_ratio` | > 2.0 sustained 24 h | `ai-quality-drift` |
| `model_version` distinct value change | Any new value | `ai-model-version-change` |

## Response Steps

1. **Receive alert** at the founder ops mailbox — `founder@ozvor.com` (Cloudflare Email Routing per `docs/runbooks/email-setup-cloudflare.md`; a dedicated `ops@ozvor.com` route may be added there). *(Updated 2026-07-10 — was `ops@organicposts.ai`, issue #213.)*
2. **Product manager** reviews the flagged `generation_id` in Supabase (admin read access): query `generation_log` where `id = {generation_id}`. Examine `prompt_user`, `model_version`, `output_text`, `prompt_system`.
3. **Assess harm type**: categorize the report as one of:
   - Hallucinated business details (factual error)
   - Stereotyping or exclusionary language (bias/fairness)
   - Harmful, defamatory, or illegal content (high severity)
   - Other quality issue
4. **Immediate mitigation** (choose the least disruptive option that addresses the risk):
   - If isolated to one generation: set `drafts.status = 'discarded'` for the affected draft via admin Supabase SQL. No code deploy required.
   - If systematic (multiple `generation_id`s showing the same pattern): pin the `model_version` in `AnthropicAdapter` to the prior known-good version via `ANTHROPIC_MODEL_VERSION_OVERRIDE` environment variable; redeploy api.
   - If a prompt injection attack is suspected: check `prompt_injection_rejected` warn logs in Axiom for elevated rate; review `INJECTION_PATTERNS` in `packages/llm/src/anthropic.ts` for gaps.
5. **Escalate to ai-ethics-reviewer** if the issue is systemic (affects >3 users or indicates a bias pattern).
6. **Customer communications**: if a user's published post contained harmful AI-generated content, notify them directly using the template below.
7. **Post-mortem**: required if the incident affects more than 3 users. File at `docs/learning/postmortems/ai-incident-YYYY-MM-DD.md`.

## Model Version Change Response (NC-1a + NC-1b)

When the `ai-model-version-change` alert fires:

1. Confirm the new `model_version` value in `generation_log`.
2. Run the A8 bias baseline against the live API (not the mock): `LIVE_API=true npx vitest run tests/ai/bias-baseline.spec.ts`. Document results in `tests/ai/bias-baseline-report.md`.
3. If the baseline passes: acknowledge the alert; update the expected version in monitoring.
4. If the baseline fails: activate the model version pin immediately. Contact Anthropic account representative.

## AI Incident Comms Template

```
Subject: Action required — AI content notice

Hi [Name],

We want to let you know that a draft generated by Ozvor for your account on [date]
may have contained [describe: inaccurate information / inappropriate language].

The draft [was / was not] published. If it was published, we recommend reviewing and
editing or removing the content.

We have corrected the underlying issue. Your [team's] content generation is not affected
going forward.

We apologize for this and take AI content quality seriously.

[Founder name]
Ozvor
```
